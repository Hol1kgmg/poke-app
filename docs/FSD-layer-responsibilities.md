# Feature-Sliced Design — 各レイヤーの責務まとめ

> 出典: https://fsd.how/ja/docs/get-started/overview/

---

## FSD とは

Feature-Sliced Design (FSD) は、フロントエンドアプリケーションの設計方法論です。コードを整理するためのルールと規約の集大成であり、**ビジネス要件が絶えず変化する中でプロジェクトをより理解しやすく、構造化されたものにすること**を目的としています。

プログラミング言語・UI フレームワーク・状態管理ライブラリを問わず、フロントエンド（Web / モバイル / デスクトップ）のアプリケーション開発に適用できます。

---

## 3つのコンセプト

### 1. レイヤー（Layers）

最上位のディレクトリ群。アプリケーション全体を「責務」ごとに縦に分割する。全 FSD プロジェクトで名前が標準化されており、**上位レイヤーは下位レイヤーのみをインポートできる**。

### 2. スライス（Slices）

レイヤー内をビジネスドメイン（機能領域）で横に分割したもの。名前は自由に決められ、**同じレイヤー内の他のスライスを参照することはできない**。

### 3. セグメント（Segments）

スライス（または App / Shared レイヤー）の中をさらに技術的な目的で分割したもの。代表的なセグメント名:

| セグメント | 内容 |
|---|---|
| `ui` | UIコンポーネント・スタイル・日付フォーマッターなど |
| `api` | バックエンドとのやりとり（リクエスト関数・型・マッパー） |
| `model` | データモデル・バリデーション・ストア・ビジネスロジック |
| `lib` | このスライス内の他モジュールが使うライブラリコード |
| `config` | 設定ファイル・フィーチャーフラグ |

---

## インポートの方向ルール

上位レイヤーは下位レイヤーのみインポートできる。同一レイヤー間・上位レイヤーへのインポートは禁止。

```
App          ← 最上位（最も不安定）
  ↓ インポート可能
Pages
  ↓
Widgets
  ↓
Features
  ↓
Entities
  ↓
Shared       ← 最下位（最も安定）
```

## 各レイヤーの責務詳細

### 🔴 App — 最上位レイヤー

アプリケーションの起動・初期化に必要なすべてのものを配置するレイヤー。

**責務:**
- ルーティング設定（React Router, Next.js App Router など）
- アプリケーションのエントリーポイント
- グローバルスタイル・テーマの定義
- グローバルプロバイダー（Context, Redux Store など）
- アナリティクスの初期化

**構成例:**
```
app/
  routes/     # ルーティング定義
  analytics/  # アナリティクス設定
  providers/  # グローバルプロバイダー
  styles/     # グローバルCSS
```

**インポートできるレイヤー:** Shared 以下すべて

> ⚠️ スライスを持たず、直接セグメントで構成される特殊なレイヤー。独自セグメントの作成が許可されている。

---

### ⚫ Processes — ~~非推奨~~

複数ページにまたがる複雑なシナリオを管理するレイヤー。**現在は非推奨（deprecated）**。

**責務:**
- 複数ページをまたぐユーザーフロー（例: 多段階の登録フロー）
- ページ間の状態管理

**構成例:**
```
processes/
  auth-flow/  # 認証フロー全体の管理
```

**インポートできるレイヤー:** Pages 以下すべて

> ⚠️ 代わりに App レイヤーや Pages レイヤーでの対応が推奨される。

---

### 🟠 Pages — 画面単位

アプリケーションの各ページ（画面）を定義するレイヤー。

**責務:**
- ページ全体の構成・レイアウト
- ネストされたルーティング時のページの大部分
- Widgets・Features・Entities を組み合わせて画面を構成する

**構成例:**
```
pages/
  home/
    ui/          # ホームページコンポーネント
  article-reader/
    ui/          # 記事読み込みページ
    api/         # ページ固有のAPIアクセス
  settings/
    ui/          # 設定ページ
```

**インポートできるレイヤー:** Widgets 以下すべて

> ページの「組み立て」を担当する。ビジネスロジックは下位レイヤーに委ねる。

---

### 🟡 Widgets — UIブロック

大きな自己完結型の機能・UI ブロックを定義するレイヤー。

**責務:**
- ユーザーシナリオ全体を実装する大きなUIブロック
- ページに埋め込まれる独立した機能ユニット（例: ヘッダー、サイドバー、フィード）
- Features・Entities を組み合わせた複合コンポーネント

**構成例:**
```
widgets/
  header/
    ui/        # ヘッダーコンポーネント
  article-feed/
    ui/        # 記事フィードUI
    model/     # フィードの状態管理
```

**インポートできるレイヤー:** Features 以下すべて

> 複数のページで共通して使われる大きな UI パーツがここに置かれる。

---

### 🟢 Features — 機能単位

ユーザーにビジネス価値をもたらすプロダクト機能の**再利用可能な**実装。

**責務:**
- ユーザーが実行できるアクション（例: いいね、コメント投稿、ログイン）
- 複数箇所で再利用されるビジネス機能
- ユーザーインタラクションに対応する処理

**構成例:**
```
features/
  auth-by-email/
    ui/        # ログインフォーム
    model/     # 認証ストア・ロジック
    api/       # 認証APIリクエスト
  like-article/
    ui/        # いいねボタン
    api/       # いいねAPIリクエスト
```

**インポートできるレイヤー:** Entities と Shared のみ

> 「再利用可能な」ことが重要。1箇所でしか使わない機能は Pages や Widgets に置くことも検討する。

---

### 🔵 Entities — ドメインモデル

プロジェクトが扱うビジネスエンティティ（ドメインオブジェクト）を定義するレイヤー。

**責務:**
- ビジネスドメインのデータモデル（例: User, Article, Comment）
- エンティティの表示コンポーネント（カード、リストアイテムなど）
- エンティティに関連するAPIリクエスト・型定義
- エンティティの状態管理

**構成例:**
```
entities/
  user/
    ui/        # ユーザーカードコンポーネント
    model/     # User の型・スキーマ
    api/       # ユーザー取得API
  article/
    ui/        # 記事カード
    model/     # Article の型・スキーマ
```

**インポートできるレイヤー:** Shared のみ

> 一度安定するとほとんど変わらないものが置かれる。ドメイン知識の核となるレイヤー。

---

### 🟣 Shared — 汎用ユーティリティ

プロジェクト・ビジネスの固有事情から切り離された再利用可能なコード。

**責務:**
- UIコンポーネントライブラリ（Button, Input, Modal など）
- 汎用ユーティリティ関数（日付変換、バリデーションなど）
- APIクライアントの設定・共通リクエスト処理
- 定数・設定値・フィーチャーフラグ
- 型定義・インターフェースの共通部分

**構成例:**
```
shared/
  ui/        # 共通UIコンポーネント
  api/       # APIクライアント設定
  lib/       # 汎用ユーティリティ
  config/    # 環境変数・設定
  i18n/      # 国際化（独自セグメントの例）
```

**インポートできるレイヤー:** なし（他のすべてのレイヤーから参照される）

> ⚠️ スライスを持たず、直接セグメントで構成される特殊なレイヤー。独自セグメントの作成が許可されている（`shared/i18n/`, `shared/icons/` など）。

---

## FSD の利点

**一貫性**
構造が標準化されているため、プロジェクトが統一され、新メンバーのオンボーディングが容易になる。

**変更・リファクタリングへの耐性**
同一レイヤーや上位レイヤーを参照できないため、予期しない副作用なしに分離した変更が可能。

**ロジックの再利用制御**
レイヤーに応じてコードを高再利用にも局所的にもできる。DRY 原則と実用性のバランスを保つ。

**ビジネス・ユーザーニーズへの焦点**
アプリケーションがビジネスドメインで分割され、命名にビジネス用語が使われるため、関係のない部分を理解せずとも有用な作業が可能。
